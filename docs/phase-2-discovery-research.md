# Phase Two Discovery Strategy for Swell Checker

## Bottom line

The strongest path forward is to build discovery **now**, but to build it as a **sidecar proposal system**, not as an automatic extension of the live watchlist/router. The current codebase already has the right safety pattern for this: score snapshots are written by default, calibration gates routing, and router outputs are emitted only as `pending_approval` until an operator approves them. The README also still describes open discovery as the deliberately deferred next phase, and the May 19 handoff identifies the open-discovery gap as the major remaining product limitation after the earlier scorer/cron issues. In other words, the repo is already conservative enough to let you ship discovery in parallel without contaminating the production watchlist, so long as discovery writes to a separate proposal queue and approved candidates go through an observation window before they join normal scoring/routing. That “ship in parallel, but gated” recommendation is an inference from the current repo design. citeturn38view1turn38view2turn37view1turn30view0

Strategically, that matches the system’s real value. The May 19 handoff frames swell-checker as the consumer-trend complement to `trend-corpus`, and specifically calls out open discovery as the missing capability that would let the product find emerging trends the operator has not already named. The same handoff also says the cleanest downstream monetization path is routing fired trends into `business-guy` so niche generation becomes data-driven instead of hand-curated. Phase two is therefore not a side quest; it is the part that turns swell-checker from a verification loop into a proprietary radar. citeturn30view0

## What the current repo already gives you

The existing repo already has most of the plumbing you need for prose-based discovery. `ingest.py` fetches “general feeds” once per cycle under a virtual `__general__` candidate, and `extract.py` already runs a separate LLM pass over those feeds. The configured general feeds include NYT Style, NYT Well, Atlantic Family, Vox Culture, Outside, Well+Good, Men’s Health, Women’s Health, Axios, and Hacker News frontpage. That means phase two does **not** need a new ingest substrate for prose discovery; it only needs a new extraction/output path. citeturn27view0turn27view1turn28view0turn10view2

But the repo is also very clear about the current limitation: the general-feed prompt explicitly tells the model to match only against the **tracked trends list**, requires `candidate_slug` from that list, and emits nothing for articles that do not map cleanly to a known candidate. Then `extract.py` skips any general-feed output whose slug is not already in the tracked slug map, increments `general_orphans`, and continues. Put plainly: the system already reads broad culture/lifestyle feeds, but novel signals are intentionally discarded today. That is the most important code-level proof that phase two is the correct next build. citeturn12view0turn7view4turn7view7

There is another important implementation detail: the current parser and sink are event-only. `parse_events()` yields only JSON objects that contain `event_type` and `evidence_quote`, and `insert_event()` writes them into the `events` table. That means you cannot get true discovery by only editing a prompt file; you need a **second parser/output sink** for proposal objects, because a proposed candidate is not an event on an existing candidate. citeturn29view1turn29view2turn34view0

The Google Trends wiring is also narrower than it first appears. The repo does have `pytrends` installed in `requirements.txt`, and `ingest.py` does call Google Trends, but only through `interest_over_time()` for a single query over `today 12-m` with `geo="US"`, producing peak value, recent/baseline averages, delta percentage, and weekly values. There is **no current use** of `related_queries()` or `trending_searches()` in the live code. So discovery is not “already there but unused”; it still requires deliberate new plumbing. citeturn4view4turn10view0turn9view2

One more thing is worth flagging before phase two starts: the repo has mild documentation drift. The public repo description says “v0: 24 curated candidates,” the README’s current-scope section says “23 trends,” the file-list comment says “22 trends in v0,” and the May 19 handoff says “Candidates seeded 24.” The README’s quick scoring reference also still says the composite is `0.4·velocity + 0.4·spread + 0.2·vocabulary`, while the live default config in `scorer.py` is spread-heavy at `0.25 / 0.55 / 0.20`. For phase two, that means the exclusion list for “known trends” and the meaning of downstream score behavior should be derived from the **database and config**, not from README prose. citeturn38view0turn38view2turn30view0turn6view7turn5view0

## Which discovery surfaces are actually worth building first

The best **free** signal is still Google Trends related/rising discovery around seed concepts. Official Google Trends help says rising searches are terms with the most significant growth in the requested period, and “Breakout” means growth above 5,000%. That is exactly the kind of adjacent-query behavior you want for unnamed or newly cohering consumer habits. `pytrends` exposes both `related_queries()` and `interest_by_region()` and documents them as wrappers over the Google Trends related-query and regional-interest surfaces. Conceptually, this is the highest-signal free surface because it finds “what is clustering around an existing habit” rather than “what is in the news today.” citeturn17view1turn21view0turn17view2

The problem is dependency risk. `pytrends` is an **unofficial** library, its own README says it is only good “until Google changes their backend,” the repo was archived in April 2025, and there are open issues for 429s, empty realtime results, and changed endpoints. Google now has an official Google Trends API alpha, but Google says it is only available to a limited set of alpha testers and describes it around consistently scaled search-interest and geographic data, not the related/rising query surfaces you want here. The consequence is not “do not use Google”; it is “build a provider abstraction.” Use `pytrends` now behind an interface, but do not hardwire phase two to it as if it were a stable permanent dependency. citeturn21view0turn20search1turn20search5turn20search8turn17view0

For the **broad breakout** Google surface, I would slightly change your proposed implementation. Official Google Trends help says the “Trending now” page covers 100+ countries, refreshes about every ten minutes, supports sorting/filtering, and can export via CSV or RSS. But Google also says “Trending now” is specifically showing search queries experiencing a recent surge that are related to a **news story**. That makes it valuable as a macro-awareness source, but by design it will be noisier and more news-heavy than seed-adjacent rising queries. My recommendation is to keep this source, but treat it as a secondary, lower-confidence surface and prefer the **official Trending Now exports** over `pytrends.trending_searches()` whenever possible. citeturn25search0turn17view2turn20search2

The **general-feed LLM pass** is the easiest and safest first build. You already ingest global prose feeds, already batch them under `__general__`, already run a Claude pass against them, and already store verbatim evidence quotes into the event system. The only missing part is allowing the model to propose a new normalized trend object instead of forcing it to attribute every valid article to a known slug. The trade-off is timing: by the time a pattern shows up in NYT Style or Men’s Health, you are later in the adoption curve than you would be on search or TikTok. But because this surface fits the existing architecture almost perfectly, it is the **lowest-risk first increment**. citeturn27view0turn27view1turn12view0turn29view5

TikTok is the highest-upside but **not** the critical path. Official TikTok developer docs show that the Research Tools can expose public account/content/shop data and that the video query API can return descriptions, hashtags, views, likes, comments, region code, voice-to-text, and other fields that would be genuinely useful for early trend detection. The catch is eligibility: TikTok says access is for qualifying researchers in eligible regions, with academic or qualifying nonprofit affiliation, independence from commercial interests, non-commercial/public-interest use, disclosed funding, and ethical review. Unless you are running this under an eligible academic/nonprofit partner, the Research API is probably not a realistic production dependency for a commercial trend product. citeturn19view0turn19view1turn19view2

That makes TikTok’s **public Creative Center** the better interim route. TikTok’s own Creative Center pages expose “Trend Discovery” across hashtags, songs, creators, and TikTok videos, and TikTok explicitly markets it as a place to discover what is buzzing in a region and what performs well. It is not as queryable or research-grade as the Research API, but it is official, public, and much lower friction. So the right ranking is: Google related/rising first, general-feed discovery second, **public TikTok Creative Center third**, and TikTok Research API only if an access path is secured later. citeturn36view0turn36view1turn36view2turn19view0

## Recommended architecture

I would keep your proposed `discover.py`, but I would **not** stop at a single `proposed_candidates` table. The better shape is a clustered proposal model with one table for the proposal entity and one for evidence. The repo already uses append-only `events` plus summary/routing layers; discovery should mirror that. A single-row proposal table will quickly become duplicate-prone because the same idea will surface as “sound bath,” “sound bathing,” “sound healing,” “gong meditation,” and so on across different sources and dates. Operator review gets much easier if the system clusters those into one proposal with multiple evidence records instead of emitting many near-duplicates. This is an architectural recommendation based on the current repo’s data model and gating pattern. citeturn34view0turn37view1

A good target shape is this:

```text
discover.py
├── adapters/
│   ├── google_related.py        # seed -> related rising / breakout queries
│   ├── google_trending_now.py   # official Trending Now RSS/CSV or scrape adapter
│   ├── general_feed_discovery.py# second LLM pass over __general__ fetches
│   └── tiktok_creative_center.py# optional interim TikTok trend surface
│
├── proposal clustering
│   ├── normalize display_name -> canonical_slug
│   ├── reject if slug already exists in candidates
│   ├── merge near-duplicates into one proposal
│   └── attach evidence rows
│
└── operator workflow
    ├── --list-pending
    ├── --approve <id>
    ├── --reject <id>
    └── --promote <id> -> candidate enters observing state
```

The key schema idea is to separate **proposal** from **evidence**. The proposal row stores `canonical_slug`, `display_name`, guessed `category`, `status`, `first_seen_at`, `last_seen_at`, `support_count`, and a compact machine explanation. The evidence rows store `surface`, `seed_slug` if relevant, source URL/reference, evidence quote, event date, raw label, geo if present, and confidence. That gives the operator something much better than “a candidate showed up”: it gives them a dossier. This is especially important because Google Trends data is sampled, normalized, and can show noise on low-volume queries, so you should not want a single low-volume anomaly to become a candidate without corroboration. citeturn26view0

You should also reuse the **router approval UX** almost verbatim. `trend_router.py` already has the right affordances: `--list-pending`, `--approve`, `--reject`, `pending_approval` state, and `requires_approval=True` in the payload. Discovery should feel the same operationally so the operator does not learn a second governance model. The code already proves that this pattern fits the repo. citeturn33view0turn33view1turn33view2turn33view4turn37view1

The biggest change I would make to your promotion flow is an explicit **observation state**. Right now, `scorer.py` scores every candidate with `status='tracking'`, and `trend_router.py` considers routable stages `{"approaching", "very_early"}` with no candidate-age or minimum-history gate. In other words, if you approve a newly discovered trend straight into normal `tracking` with `stage='very_early'`, the existing system can score it and potentially route it almost immediately if early signals are strong enough. If you want a four-week observation window, that needs to be enforced in data/state, not just remembered operationally. I would add either `status='observing'` plus a later transition to `tracking`, or a `watchlist_eligible_at` / `router_eligible_at` field that the scorer/router respect. citeturn37view0turn37view1

I would **not** make discovery wait on extending the current calibration suite. `calibration.py` only evaluates a fixed expectation set of known positives and negatives — pickleball, hyrox, axe_throwing, and crossfit — and that is exactly what it should continue doing. Discovery needs its **own** quality loop, centered on proposal precision, duplicate suppression, corroboration rate, and later-stage validation, not on the phase-one calibration assertions. So the safest split is: existing calibration continues to guard routing for tracked candidates, while discovery gets a separate acceptance harness before promotion. citeturn13view0turn13view1turn13view4

## Rollout sequence and success metrics

The smallest safe rollout is to start with **general-feed discovery + orphan capture**. Because the current system already reads the feeds and already discards off-list discoveries as `general_orphans`, this stage buys you immediate product value with the least engineering risk. Add a second prompt, new parser, proposal clustering, and operator review UI. Do not auto-promote anything yet. citeturn7view4turn7view7turn27view0

Then add the **Google discovery provider**. Start with seed-adjacent related/rising queries, because they are the most structurally aligned with your objective of finding “next-to-known” consumer behaviors. Add Trending Now only after that, and treat it as a recall-oriented broad net. If possible, pull Trending Now from the official exports rather than the old `pytrends.trending_searches()` method. Given the fragility of `pytrends`, the provider layer should record adapter failures without killing the rest of discovery. citeturn17view1turn25search0turn21view0turn20search10

After that, wire in **promotion and observation gating**. Approval should do three things at once: create the candidate, generate default source seeds, and mark the candidate as observing rather than fully routable. For source creation, I would start newly approved candidates with Google Trends plus general-feed coverage, and add Reddit only after approval rather than at proposal time. That keeps ingest cost and noise bounded. The repo’s six-hour cron loop is already sufficient cadence for this; there is no need to turn discovery into a minute-by-minute stream just because some upstream sources refresh that quickly. citeturn38view1turn27view1

Only then would I add the **TikTok layer**, and only in two steps: public Creative Center first, Research API later if — and only if — you have a compliant access path. That sequencing matters because the official eligibility constraints make Research API access uncertain for a commercial product, while Creative Center is available now. citeturn19view0turn36view0

The metrics that matter most are not raw proposal volume; they are **precision and downstream validity**. The right dashboard is: operator approval rate, duplicate/merge rate, median evidence count per approved proposal, percentage of approved proposals that later accumulate spread/operator evidence, and percentage of approved proposals that are still interesting after the observation window. That last one matters particularly because the current scorer is deliberately spread-heavy in code, so what you ultimately want from discovery is not just chatter but proposals that later develop into the kinds of signals the live model already rewards. citeturn5view0turn6view7

## Risks and open questions

The biggest technical risk is **Google dependency fragility**. `pytrends` is archived and brittle, while Google’s official API is currently alpha-limited and, from the published announcement, is focused on search-interest series and geography rather than related/rising query discovery. The mitigation is straightforward: use a provider abstraction from day one and avoid coupling core proposal logic to one unofficial client. citeturn21view0turn17view0turn20search8

The biggest product risk is **over-promotion** of weak ideas. Google Trends is sampled and normalized; low-volume terms can show statistical noise; Trending Now is explicitly tied to news surges; and general feeds are often later-stage validation rather than early discovery. That is why clustered multi-surface evidence plus an observation state matters more than squeezing every possible candidate into the watchlist as early as possible. citeturn26view0turn25search0turn12view0

The biggest operational risk is **governance drift** if discovery invents a second approval system. The repo already has a conservative pattern for pending approvals and operator execution gates, so discovery should reuse that pattern instead of inventing a separate review tool. citeturn38view2turn37view1

There are also two genuine open questions. The first is architectural: the May 19 handoff still leaves open whether swell-checker should remain a standalone codebase or later converge toward a `trend-corpus`-style runtime/export model. For phase two specifically, my recommendation is to stay in-repo and ship discovery in place first, then export or mirror after the discovery queue is proven. The second is operational: I have grounded this report in the **public repo code** and the **May 19 handoff/status snapshot**, but I did not verify a live database or current production runtime, so any post-May changes in data freshness, cron health, or candidate counts could alter tactical details even if they do not change the architectural recommendation. citeturn30view0turn38view0
